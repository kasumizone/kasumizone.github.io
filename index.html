二、基本原理

Spring Security 本质是一个过滤器链

1. 过滤器

FilterSecurityInterceptor

方法级的权限过滤器，基本位于过滤连的最底部

ExceptionTranslationFilter

异常过滤器，用于处理在认证授权过程中抛出的异常

UsernamePasswordAuthenticationFilter

对于用户名，密码进行校验的过滤器

…

1.1 过滤器的加载过程

DelegatingFilterProxy

这个过滤器中的doFilter方法：

doFilter方法中执行的init方法

该方法调用了FilterChainProxy过滤器

FilterChainProxy

在doFilter方法中调用doFilterInternal方法

doFilterInternal方法中有getFilters方法

getFilters方法，得到所有的过滤器，加载到过滤连中

getFilters方法是SecurityFilterChain接口中定义的

以上过程，非Spring Boot项目则需要手动进行

2. 两个接口

UserDetailsService

实现该接口进行数据库中用户名密码的查询

实际项目中，我们需要查询数据库，得到用户信息，做登录验证，而不是用Security框架提供的默认用户名和随机密码，所以需要：

  • 自定义一个过滤器类，继承UsernamePasswordAuthenticationFilter类

    重写其中的3个方法：

    • attemptAuthentication：得到请求中的用户名和密码，进行验证

    和它的父类AbstractAuthenticationProcessingFilter中的：

    • successfulAuthentication：验证成功

    • unsuccessfulAuthentication：验证失败

  • 下面需要写查询数据库中的用户名和密码并返回数据的方法

    写一个类实现UserDetailsService接口，编写查询数据库的过程，返回User对象，这个User对象是Security框架提供的对象。

UserDetailsService接口中loadUserByUsername方法的返回值

实现UserDetailsService接口需要重写loadUserByUsername方法，在这里面根据用户名进行操作，

这个方法的返回值类型是UserDetails，可以自定义一个类，包含用户名，密码，权限等信息，去实现这个接口

同时，Security框架提供了一个User类，实现了UserDetails接口，所以可以直接返回User对象即可

User.class中的有参构造方法：

//用户名，密码，权限集合
public User(String username, String password, Collection<? extends GrantedAuthority> authorities) {
    this(username, password, true, true, true, true, authorities);
}

PasswordEncoder

数据加密的接口，用于返回User对象中密码的加密

BCryptPasswordEncoder类实现了该接口

三、Web权限方案

1. 认证

设置登录的用户名和密码

1.1 三种认证方式

1 通过application.properties/yaml配置文件进行配置

#设置用户名
spring.security.user.name=Alice
#设置密码
spring.security.user.password=123456

这样进行配置，启动项目将不再默认生成随机密码，即可使用配置文件中的用户名和密码进行登录

2 通过配置类进行配置

新建一个配置类，继承WebSecurityConfigurerAdapter，重写其中的configure方法，

进行配置用户名，密码和角色

@Configuration
public class SecurityConfig extends WebSecurityConfigurerAdapter {
    @Override
    protected void configure(AuthenticationManagerBuilder auth) throws Exception {
        BCryptPasswordEncoder passwordEncoder = new BCryptPasswordEncoder();
        String password = passwordEncoder.encode("123123");
        auth.inMemoryAuthentication().withUser("lover").password(password).roles("admin");
    }

    //需要创建PasswordEncoder对象，否则会报错
    @Bean
    PasswordEncoder password(){
        return new BCryptPasswordEncoder();
    }
}

实际情况下，用户名和密码不会是固定的某个值，而是从数据库中查询，所以以上两种方式均不常用

3 自定义编写实现类（实现UserDetailsService接口）

Security认证过程中，会先查找配置文件和配置类中有没有进行用户名和密码的配置，

如果找不到，则会找UserDetailsService接口

第一步：创建配置类，设置使用哪一个UserDetailsService接口的实现类

@Configuration		//注入到配置类中
public class SecurityUserConfig extends WebSecurityConfigurerAdapter {

    @Autowired
    UserDetailsService userDetailsService;

    @Override
    protected void configure(AuthenticationManagerBuilder auth) throws Exception {
        //指定使用哪个UserDetailsService实现类
        auth.userDetailsService(userDetailsService).passwordEncoder(password());
    }

    //需要创建PasswordEncoder对象，否则会报错
    @Bean
    PasswordEncoder password(){
        return new BCryptPasswordEncoder();
    }
}

第二步，编写实现类，返回User对象，User对象中有用户名，密码和操作权限

查询数据库，得到用户名，密码和操作权限，封装到User对象中，进行返回

固定用户名密码（仅供测试，不实用）

@Service("userDetailsService")
public class MyUserDetailsService implements UserDetailsService {

第四步，创建Mapper层，继承BaseMapper

第五步，在UserDetailService接口的实现类中编写根据用户名查询用户名，密码和角色权限等信息的流程，

并返回User对象

@Service("userDetailsService")
public class MyUserDetailsService implements UserDetailsService {

    //调用Mapper中的方法
    @Autowired
    private UsersMapper userMapper;

    // String s 为表单传来的用户名
    @Override
    public UserDetails loadUserByUsername(String s) throws UsernameNotFoundException {
        //根据用户名s，电泳UserMapper，查询数据库
        QueryWrapper<Users> wrapper = new QueryWrapper<>();
        wrapper.eq("username",s);
        Users users = userMapper.selectOne(wrapper);
        //判断
        //数据库中没有用户名，则认证失败
        if (users==null){
            throw new UsernameNotFoundException("用户名不存在");
        }

        //模拟创建一个集合，使用其中的工具类，加角色
        List<GrantedAuthority> auths = AuthorityUtils.commaSeparatedStringToAuthorityList("role");
        //从查询数据库返回的的users对象中，得到用户名和密码进行返回
        return new User(users.getUsername(),new BCryptPasswordEncoder().encode(users.getPassword()),auths);
    }
}

2. 自定义登录页面和设置不需要认证的请求

配置类中实现

@Override
protected void configure(HttpSecurity http) throws Exception {
    //自定义自己编写的登录页面
    http.formLogin()
            .loginPage("/login.html")           //登陆页面设置
            .loginProcessingUrl("/user/login")  //提交登陆后访问的Controller路径，这个Controller不需要我们编写，Security框架已经做到了，只需要随便给定一个路径即可
            .defaultSuccessUrl("/test/index").permitAll()   //登录成功后，要跳转到的路径
            .and().authorizeRequests()
                .antMatchers("/","/test/hello","/user/login").permitAll() //设置访问的哪些路径不需要认证
            .anyRequest().authenticated()
            .and().csrf().disable();            //关闭csrf防护
}

2. 授权

基于角色或权限进行访问控制

2.1 hasAuthority 方法

如果当前的主体具有指定的权限，则返回 true,否则返回 false

比如访问的/index需要用管理员的角色或权限才能访问，

那么这个用户有该角色或权限，则返回true，否则返回false

1 在配置类中设置当前访问地址需要有哪些权限才能访问

.and().authorizeRequests()
                    //该路径只有具有 admins 权限的用户才能访问
                    .antMatchers("/test/index").hasAuthority("admins")

2 在UserDetailsService的实现类中给返回的User对象设置权限

//模拟创建一个集合，使用其中的工具类，加角色
List<GrantedAuthority> auths = AuthorityUtils.commaSeparatedStringToAuthorityList("admins");

如果没有权限，则会报 403 错误 (type=Forbidden, status=403)

hasAuthority 方法只能做到单个权限的访问限制，多个需要用到hasAnyAuthority 方法

2.2 hasAnyAuthority 方法

如果当前的主体有任何提供的角色（给定的作为一个逗号分隔的字符串列表）的话，返回 true

比如当前访问的/index需要有admins，manager任意一个权限就能够访问，

如果用户有其中任意一个权限，则返回true，否则返回false

配置类中配置如下

.and().authorizeRequests()
    //该路径具有 admins,manager 中任意权限的用户才能访问
    .antMatchers("/test/index").hasAnyAuthority("admins,manager")

2.3 hasRole 方法

如果当前主体具有指定的角色，则返回 true

hasRole方法的源码中，在role前面有ROLE_，所以在给用户赋予角色时，需要加上ROLE_

配置类中配置如下

.antMatchers("/test/index").hasRole("sale")

在UserDetailsService的实现类中给返回的User对象设置角色

//模拟创建一个集合，使用其中的工具类，加角色
List<GrantedAuthority> auths = AuthorityUtils.commaSeparatedStringToAuthorityList("admins,ROLE_sale");

2.4 hasAnyRole 方法

表示用户具备任何一个条件都可以访问

.antMatchers("/test/index").hasAnyRole("sale","admin")

2.5 自定义403没有权限访问的页面

配置类中写如下配置

protected void configure(HttpSecurity http) throws Exception {
    ...;
    //配置没有权限访问跳转的自定义页面
    http.exceptionHandling().accessDeniedPage("/unauth.html");
    ...;
}

3. 注解使用

3.1 @Secured

**判断用户是否具有指定角色，另外需要注意的是这里匹配的字符串需要添加前缀”ROLE_“

1 在启动类/配置类上添加注解@EnableGlobalMethodSecurity(securedEnabled=true)开启注解功能

public class SpringSecurityDemo1Application {
    public static void main(String[] args) {
        SpringApplication.run(SpringSecurityDemo1Application.class, args);
    }
}

2 在Controller方法上使用注解，设置角色

@Secured({"ROLE_sale","ROLE_manager"})
@GetMapping("update")
public String update(){
    return "hello update!";
}

3.2 @PreAuthorize

在进入方法之前进行验证，看有没有权限进行访问

1 启动类/配置类开启注解

@EnableGlobalMethodSecurity(prePostEnabled = true)

2 在Controller方法上使用注解

@PreAuthorize("hasAnyAuthority('admins')")
@GetMapping("update")
public String update(){
    return "hello update!";
}

3.3 @PostAuthorize

在方法执行之后进行校验（不常用）

1 启动类/配置类开启注解

@EnableGlobalMethodSecurity(prePostEnabled = true)

2 在Controller方法上使用注解

@PostAuthorize("hasAnyAuthority('admins')")
@GetMapping("update")
public String update(){
    System.out.println("update......");
    return "hello update!";
}

因为是在方法执行之后进行校验，所以如果没有权限访问，也会执行控制台输出update……的代码

3.4 @PostFilter

对方法返回的数据进行过滤（不常用）

权限验证之后对数据进行过滤 留下用户名是 admin1 的数据

表达式中的 filterObject 引用的是方法返回值 List 中的某一个元素

@RequestMapping("getAll")
@PreAuthorize("hasRole('ROLE_管理员')")
@PostFilter("filterObject.username == 'admin1'")
@ResponseBody
public List<UserInfo> getAllUser(){
    ArrayList<UserInfo> list = new ArrayList<>();
    list.add(new UserInfo(1l,"admin1","6666"));
    list.add(new UserInfo(2l,"admin2","888"));
    return list;
}

3.5 @PreFilter

对传入方法的数据进行过过滤（不常用）

@RequestMapping("getTestPreFilter")
@PreAuthorize("hasRole('ROLE_管理员')")
@PreFilter(value = "filterObject.id%2==0")
@ResponseBody
public List<UserInfo> getTestPreFilter(@RequestBody List<UserInfo>
                                       list){
    list.forEach(t-> {
        System.out.println(t.getId()+"\t"+t.getUsername());
    });
    return list;
}

import com.alibaba.cloud.nacos.NacosDiscoveryProperties;
import com.alibaba.cloud.nacos.ribbon.NacosServer;
import com.alibaba.nacos.api.exception.NacosException;
import com.alibaba.nacos.api.naming.NamingService;
import com.alibaba.nacos.api.naming.pojo.Instance;
import com.netflix.client.config.IClientConfig;
import com.netflix.loadbalancer.AbstractLoadBalancerRule;
import com.netflix.loadbalancer.BaseLoadBalancer;
import com.netflix.loadbalancer.ILoadBalancer;
import com.netflix.loadbalancer.Server;
import lombok.extern.slf4j.Slf4j;
import org.springframework.beans.factory.annotation.Autowired;

@Slf4j
public class NacosWeightedRule extends AbstractLoadBalancerRule {

    @Autowired
    private NacosDiscoveryProperties nacosDiscoveryProperties;

    @Override
    public void initWithNiwsConfig(IClientConfig iClientConfig) {
        //读取配置文件
    }

    @Override
    public Server choose(Object o) {
        ILoadBalancer loadBalancer = this.getLoadBalancer();
        BaseLoadBalancer baseLoadBalancer = (BaseLoadBalancer) loadBalancer;
        //获取要请求的微服务名称
        String name = baseLoadBalancer.getName();
        //获取服务发现的相关API
        NamingService namingService = nacosDiscoveryProperties.namingServiceInstance();
        try {
            Instance instance = namingService.selectOneHealthyInstance(name);
            log.info("选择的实例是port={},instance={}",instance.getPort(),instance);
            return new NacosServer(instance);
        } catch (NacosException e) {
            e.printStackTrace();
            return null;
        }
    }
}

三、Sentinel 服务限流降级

1. 降级，限流，熔断

降级：访问量大的出现，限制一些功能的使用，只提供部分功能

        比如之前能提供5个功能，限制访问的太多，只给你提供三个功能，来减轻压力

限流：比如规定10个访问量，第11个就进不来，给你抛异常

熔断：比如要调用别人，对方出现故障，则熔断不允许访问

降级和限流是应对自身系统的故障，熔断是应对外部系统的故障

package lover.alice.config;

import com.alibaba.csp.sentinel.adapter.servlet.callback.RequestOriginParser;
import org.springframework.util.StringUtils;

import javax.servlet.http.HttpServletRequest;

public class RequestOriginParserDefinition implements RequestOriginParser {
    @Override
    public String parseOrigin(HttpServletRequest httpServletRequest) {
        String name = httpServletRequest.getParameter("name");
        if(StringUtils.isEmpty(name)){
            throw new RuntimeException("name is null");
        }
        return name;
    }
}

要让 RequestOriginParserDefinition 生效，需要在配置类中进行配置。

@Configuration
public class SentinelConfiguration {

    @PostConstruct
    public void init(){
        WebCallbackManager.setRequestOriginParser(new RequestOriginParserDefinition());
    }
}

一眼看去，Swagger的请求比浏览器发出的要长一些，仔细看就会发现，Swagger发送的请求，将参数中的所有特殊字符按照相应规则进行了转义编码，发现特殊字符不只是有‘/’，只是‘/’连Swagger也无法进行判断帮你转义。

1.1 主键生成策略

雪花算法：

Twitter的snowflake算法 几乎保证全球唯一

snowflake是Twitter开源的分布式ID生成算法，结果是一个long型的ID。其核心思想是：使用41bit作为毫秒数，10bit作为机器的ID（5个bit是数据中心，5个bit的机器ID），12bit作为毫秒内的流水号（意味着每个节点在每毫秒可以产生 4096 个 ID），最后还有一个符号位，永远是0。具体实现的代码可以参看https://github.com/twitter/snowflake。

不勾选自增，也不配置TableId 的话，默认是唯一ID

4. 逻辑删除(deleted)

物理删除：从数据库中直接删除

逻辑删除：在数据库中仍然存在，通过改变一个值让他失效deleted=0 ==> deleted=1

管理员可以查看被删除的记录，类似于回收站

四、乐观锁（version）(旧版本)

乐观锁 : 故名思意十分乐观，它总是认为不会出现问题，无论干什么不去上锁！如果出现了问题， 再次更新值测试

悲观锁：故名思意十分悲观，它总是认为总是出现问题，无论干什么都会上锁！再去操作！

乐观锁的实现方式：

* 取出记录时，获取当前version属性的值
* 更新时，检查version属性的值是否符合要求，如果否，则不执行更新操作
* 执行更新时，对version值进行修改

测试：

  1. 给数据库中加version字段，默认值为1

  1. 实体类加上对应字段并标注@Version注解
@Version   //乐观锁version注解
private Integer version;
  1. 注册组件(config包)
@Configuration //配置类
public class MybatisPlusConfig {
//    注册乐观锁插件
    @Bean
    public OptimisticLockerInterceptor optimisticLockerInterceptor() {
        return new OptimisticLockerInterceptor();
    }

}
  1. 测试
@Test
   void testop(){
       /**
        * 1.查询用户信息
        */
       User user = userMapper.selectById(1);
	//  2.修改用户信息
       user.setName("leguan2");
       user.setEmail("leguan2@ll.com");
       //3.执行更新操作
       userMapper.updateById(user);
   }
   // 测试乐观锁失败！多线程下
    @Test
    public void testOptimisticLocker2(){
// 线程 1
        User user = userMapper.selectById(1L);
        user.setName("kuangshen111");
        user.setEmail("24736743@qq.com");
// 模拟另外一个线程执行了插队操作
        User user2 = userMapper.selectById(1L);
        user2.setName("kuangshen222");
        user2.setEmail("24736743@qq.com");
        userMapper.updateById(user2);
// 自旋锁来多次尝试提交！
        userMapper.updateById(user); // 如果没有乐观锁就会覆盖插队线程的值！
    }

总结

多线程操作同一个对象的情况下，在第一个线程还没有更新的情况下，插队线程先修改并完成更新，如果没有乐观锁的话，那么第一个线程会覆盖掉第二个插队线程修改的值，有了乐观锁，第一个线程就不会覆盖插队线程修改的值，因为插队线程修改时，已经将乐观锁的version值+1了，第一个线程在进行修改是判断version的值与条件不符，所以将不会执行修改操作。

//按条件分页查询课程列表
@Override
public IPage<CoursePublishVo> getCourseDisplayListService(Integer current, Integer limit, CourseQuery courseQuery) {

    Page<CoursePublishVo> page = new Page<>(current,limit);

    QueryWrapper<CourseQuery> wrapper = new QueryWrapper<>();
    String title = courseQuery.getTitle();
    String teacherName = courseQuery.getTeacherName();
    String subjectParentName = courseQuery.getSubjectParentName();
    String subjectName = courseQuery.getSubjectName();
    if (!StringUtils.isEmpty(title)){
        wrapper.like("ec.title",title);
    }
    if (!StringUtils.isEmpty(teacherName)){
        wrapper.eq("ec.name",teacherName);
    }
    if (!StringUtils.isEmpty(subjectParentName)){
        wrapper.eq("es1.title",subjectParentName);
    }
    if (!StringUtils.isEmpty(subjectName)){
        wrapper.eq("es2.title",subjectName);
    }
    IPage<CoursePublishVo> coursePage = baseMapper.getCourseDisplayList(page, wrapper, current, limit, courseQuery);
    return coursePage;
}
